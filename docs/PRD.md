# LinkU 产品需求文档 PRD

## 1. 产品背景

大学生之间存在大量高频但分散的资源匹配场景，例如找家教、当家教、找饭搭子、自习搭子、运动搭子、黑客松队友、保研考研经验咨询、简历修改和项目辅导等。

传统校园论坛、微信群、QQ群和表白墙存在几个明显问题：

- 信息噪音高，用户需要在大量无关消息中搜索。
- 需求表达不标准，筛选成本高。
- 发布后等待回复，反馈链路慢。
- 信任机制弱，容易出现骚扰、诈骗、联系方式泄露和线下安全风险。
- 内容形态偏帖子流，不适合轻量快速判断“这个人是否合适”。

LinkU 希望用 AI 和滑卡交互重做校园资源匹配：用户只输入一句自然语言需求，系统解析需求、生成卡片、推荐候选对象，并通过左右滑和双向匹配降低沟通成本。

## 2. 产品目标

- 帮大学生快速匹配家教、搭子和高价值经验资源。
- 降低需求发布成本，让用户不必学习复杂表单。
- 降低筛选成本，用卡片、匹配度、标签和 AI 推荐理由辅助判断。
- 用 AI 提升建档、卡片生成、破冰、审核和安全提示效率。
- 保持校园友好和安全边界，不把产品做成低俗 dating app。
- 为后续会员、Premium Match、咨询抽成和优先推荐预留商业化空间。

## 3. 用户角色

| 用户角色       | 核心诉求                                             |
| -------------- | ---------------------------------------------------- |
| 普通学生       | 浏览卡片、滑卡、匹配、联系、举报、拉黑               |
| 找家教的人     | 按科目、价格、时间、学校、线上/线下匹配家教          |
| 提供家教的人   | 展示擅长科目、价格、可约时间、认证和评价             |
| 找搭子的人     | 找饭搭子、自习搭子、运动搭子、电影搭子、黑客松队友等 |
| 经验咨询提供者 | 提供保研考研、简历、项目、竞赛、AI coding 陪跑等服务 |
| 管理员/审核员  | 审核卡片、处理举报、封禁用户、查看 AI 审核结果       |

## 4. 核心使用路径

1. 用户进入 LinkU。
2. MVP 阶段使用 mock login 登录。
3. 用户选择专区：家教专区、搭子专区、学长学姐专区。
4. 用户创建个人资料，填写学校、城市、年级、专业、技能、兴趣和关系边界。
5. 用户输入一句自然语言需求。
6. AI Link 解析需求并生成结构化字段。
7. AI Link 生成卡片文案。
8. 用户预览并确认发布卡片。
9. 系统根据规则评分推送候选卡片。
10. 用户左滑跳过或右滑想联系。
11. 双方互相右滑后生成匹配。
12. 匹配成功页展示匹配理由、安全提示和 AI 破冰话术。
13. 用户发起联系申请。
14. 用户可以举报、拉黑或评价。
15. 管理员在后台处理举报和审核内容。

## 5. 三大专区需求

### A. 家教专区

MVP 重点实现。

核心场景：

- 找家教：高数、英语、计算机、考研、竞赛、编程等。
- 当家教：展示科目、经验、价格、时间和认证状态。

关键字段：

- 需求类型：找家教 / 当家教。
- 科目或技能。
- 学校、城市、校区。
- 预算区间或报价区间。
- 可用时间。
- 线上、线下或混合。
- 认证状态。
- 评分和评价。

### B. 搭子专区

MVP 重点实现。

核心场景：

- 饭搭子。
- 自习搭子。
- 运动搭子。
- 台球搭子。
- 电影搭子。
- 黑客松队友。
- 英语口语搭子。
- 轻社交。

安全边界：

- 必须设置关系边界。
- 可以保留“可发展恋爱”的偏好选项，但不能作为主定位。
- 不允许低俗、擦边、骚扰和性暗示匹配。
- 双向匹配后才允许联系申请。

### C. 学长学姐专区 / Premium Match

MVP 先做展示和 mock，不接真实支付。

核心场景：

- 保研经验咨询。
- 考研经验咨询。
- 简历修改。
- 项目修改。
- AI coding 陪跑。
- 竞赛经验。
- 高校资源匹配。

命名边界：

- 不叫“985 付费专区”。
- 使用“学长学姐专区”“高价值专区”“Premium Match”“经验咨询专区”等表达。
- 不做学历歧视表达，不暗示学校层级决定人的价值。

## 6. AI Link 功能

| 功能            | MVP 说明                                                                          |
| --------------- | --------------------------------------------------------------------------------- |
| AI 需求解析     | 把自然语言解析为 zone、intent、subject、skills、budget、schedule、location 等字段 |
| AI 用户画像生成 | 根据资料和需求生成标签摘要，MVP 可简化                                            |
| AI 卡片文案生成 | 生成 title、subtitle、description、tags、highlight                                |
| AI 推荐理由生成 | 根据后端匹配分和命中维度生成一句自然语言解释                                      |
| AI 破冰话术生成 | 匹配成功后生成 3 条礼貌、低压力话术                                               |
| AI 内容审核     | 审核用户输入、卡片和联系申请                                                      |
| AI 风险提示     | 对线下见面、付费、联系方式泄露给出提示                                            |

AI 不直接决定封禁、支付、匹配安全和最终审核结论。后端规则和管理员操作必须兜底。

## 7. 信任与安全机制

- 校园邮箱认证：MVP 先 mock，后续接真实学校邮箱。
- 昵称展示：默认不直接展示真实姓名。
- 双向匹配：双方右滑后才能发起联系申请。
- 举报和拉黑：MVP 必做。
- 敏感内容审核：规则审核 + AI 审核。
- 线下见面安全提示：在匹配成功和联系申请处展示。
- 付费专区资质认证：MVP 使用 mock verified 状态。
- 评价系统：MVP 可做轻量评分，复杂评价后续完善。

## 8. 商业化设想

- 免费用户每日滑卡限制。
- 免费用户每日右滑次数限制。
- 学期会员。
- 查看谁喜欢我。
- 高级筛选。
- Premium Match 咨询抽成。
- 优先推荐。

MVP 阶段只做会员和付费状态 mock，不接真实支付。

## 9. MVP 范围

必须完成：

- mock login + JWT。
- 三大专区入口。
- 用户资料创建。
- 自然语言需求输入。
- AI 或 fallback 需求解析。
- AI 或 fallback 卡片生成。
- 卡片式推荐。
- 左滑和右滑。
- 双向匹配。
- 匹配成功页面。
- 联系申请。
- 举报和拉黑。
- 管理后台基础审核。

重点实现：

- 家教专区。
- 搭子专区。

轻量展示：

- 学长学姐专区 / Premium Match。
- 会员 / 付费 mock。

## 10. 非 MVP 暂缓功能

- 完整支付系统。
- 复杂订单和退款。
- 真正机器学习推荐模型。
- 多端 App。
- 完整 IM 长连接。
- WebSocket 聊天。
- 大规模数据分析后台。
- 复杂校园组织权限体系。
